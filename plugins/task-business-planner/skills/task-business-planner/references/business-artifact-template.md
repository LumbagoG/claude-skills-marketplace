# Business Draft Artifact Template

Шаблон артефакта для бизнес-задач. Создаётся в `wiki/tasks/YYYY-MM-DD-NNN-short-name.md` после того как партнёр ответил на вопросы. **Все поля пишутся в продуктовом языке — никаких технических деталей.**

## Ключевое отличие от технического шаблона

| Поле | Бизнес-черновик | Технический план |
|------|-----------------|-------------------|
| `status` | `business-draft` | `pending` / `in-progress` |
| `ready_for_dev` | `false` | `true` |
| `created_by` | `business-partner` | (не нужно) |
| Раздел «Технический план» | пустой, помечен «*Заполняется разработчиком*» | заполнен полностью |
| Раздел «Фазы выполнения» | отсутствует | таблица 8–14 фаз |
| Раздел «Схема данных» | отсутствует | Prisma-модели |

## Полный шаблон

````markdown

type: task

## Как выглядит (UX словами)

> Описание интерфейса в обычной речи. Не пиши как «Добавляется компонент <BulkMessageForm/>» — пиши «На странице появляется новый блок».

### Где это живёт

<Раздел в меню / страница / виджет / модальное окно.>

Пример: *«В админке тренера (левое меню) появляется новый пункт 'Рассылки'. На этой странице — список прошлых рассылок (если были) и кнопка 'Новая рассылка'.»*

### Поток действий

1. <Шаг 1>
2. <Шаг 2>
3. <Шаг 3>
4. <Шаг 4>

Пример:
1. *Тренер заходит в раздел 'Рассылки' и нажимает 'Новая рассылка'.*
2. *Открывается окно с полем для текста сообщения и кнопкой 'Прикрепить картинку'.*
3. *Тренер пишет текст, опционально прикрепляет картинку, видит превью того, как сообщение увидят ученики.*
4. *Жмёт 'Отправить всем'. Появляется подтверждение — список учеников и финальная кнопка 'Подтвердить'.*
5. *После подтверждения — сообщение «Отправлено N ученикам», возврат на страницу 'Рассылки'.*

### Как это получают пользователи

<Email / push / в приложении / SMS / комбинация.>

Пример: *«Ученики получают сообщение тремя способами: email + красная точка с числом непрочитанных в приложении + если они онлайн — всплывающее уведомление. Тренер видит для каждого получателя, прочитал он сообщение или нет.»*

### Аналогии (если применимо)
````

## Workflow обновления `wiki/tasks/index.md`

Когда создаёшь новый business-draft, в индексе задач добавь его в **отдельную секцию**, а не в общую очередь.

### Шаблон секции (создавай если ещё нет)

```markdown
## 🔮 Бизнес-черновики

> Задачи от бизнес-партнёра, ждущие технической доработки от разработчика.
> При взятии в работу скилл task-planner автоматически дополнит каждую задачу техническим планом.

| ID | Задача | Создана | Файл |
|----|--------|---------|------|
| NNN | <название> | YYYY-MM-DD | [[YYYY-MM-DD-NNN-name]] |
```

Эта секция размещается **между** «🟡 Очередь» и «🔴 Заблокированы» — визуально и логически черновики стоят перед готовыми к работе задачами.

### НЕ добавляй в «🟡 Очередь»

Очередь — для готовых к разработке задач. Бизнес-черновик не готов: разработчик может начать работу не по тому пути, не задав себе технические вопросы. Ставь только в «🔮 Бизнес-черновики».

После того как разработчик дополнит план и сменит статус на `pending`, **task-planner сам переместит задачу из черновиков в очередь**.

## Что ОБЯЗАТЕЛЬНО должно быть в артефакте

Минимальный набор, без которого артефакт не считается полным:

- [ ] Verbatim промпт партнёра
- [ ] Главная персона
- [ ] Хотя бы одна метрика успеха ИЛИ явный «открытый вопрос: метрика»
- [ ] Хотя бы один сценарий использования
- [ ] Описание UX словами (поток действий)
- [ ] Приоритет
- [ ] Status: `business-draft`
- [ ] ready_for_dev: false
- [ ] Запись в `wiki/tasks/index.md` в секции «Бизнес-черновики»

Если что-то из этого отсутствует — задай ещё ОДИН вопрос партнёру или зафиксируй как «открытый вопрос», но не оставляй пустое поле.
